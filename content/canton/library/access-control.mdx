Access Control provides role-based authorization for Daml workflows. It lets a contract define distinct roles for distinct privileged actions, rather than concentrating every privileged action behind a single party.

## Overview

Role-based access control (RBAC) associates each privileged action with a role, and each role with a set of members. An admin role governs which parties may grant and revoke membership of the other roles. This mirrors the model many teams already know from OpenZeppelin's `AccessControl` on other ecosystems, adapted to Daml's authorization and contract model.

## When to use it

Reach for Access Control when more than one party, or more than one kind of privileged action, needs to be authorized independently: for example separating a minter, a pauser, and an upgrader. When a single owner is enough, [Ownable](/canton/library/ownable) is the simpler choice.
