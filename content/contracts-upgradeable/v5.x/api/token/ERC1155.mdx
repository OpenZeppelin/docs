This set of interfaces and contracts are all related to the [ERC-1155 Multi Token Standard](https://eips.ethereum.org/EIPS/eip-1155).

The ERC consists of three interfaces which fulfill different roles, found here as [`IERC1155`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155), [`IERC1155MetadataURI`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155MetadataURI) and [`IERC1155Receiver`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver).

[`ERC1155`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#ERC1155) implements the mandatory [`IERC1155`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155) interface, as well as the optional extension [`IERC1155MetadataURI`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155MetadataURI), by relying on the substitution mechanism to use the same URI for all token types, dramatically reducing gas costs.

Additionally there are multiple custom extensions, including:

* designation of addresses that can pause token transfers for all users ([`ERC1155Pausable`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#ERC1155Pausable)).
* destruction of own tokens ([`ERC1155Burnable`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#ERC1155Burnable)).

<Callout>
This core set of contracts is designed to be unopinionated, allowing developers to access the internal functions in ERC-1155 (such as [`_mint`](#ERC1155-_mint-address-uint256-uint256-bytes-)) and expose them as external functions in the way they prefer.
</Callout>

*public*

`interfaceId`. See the corresponding

<a id="ERC1155Upgradeable-uri-uint256-"></a>

### `uri(uint256) ‚Üí string`
*public*

See [`IERC1155MetadataURI.uri`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155MetadataURI-uri).

This implementation returns the same URI for *all* token types. It relies
on the token type ID substitution mechanism
[defined in the ERC](https://eips.ethereum.org/EIPS/eip-1155#metadata).

Clients calling this function must replace the `\id\` substring with the
actual token type ID.

<a id="ERC1155Upgradeable-safeTransferFrom-address-address-uint256-uint256-bytes-"></a>

### `safeTransferFrom(address from, address to, uint256 id, uint256 value, bytes data)`
*public*

Transfers a `value` amount of tokens of type `id` from `from` to `to`.

<Callout type='warn'>
This function can potentially allow a reentrancy attack when transferring tokens
</Callout>
to an untrusted contract, when invoking [`IERC1155Receiver.onERC1155Received`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155Received) on the receiver.
Ensure to follow the checks-effects-interactions pattern and consider employing
reentrancy guards when interacting with untrusted contracts.

Emits a [`TransferSingle`](https://docs.openzeppelin.com/contracts/api/utils#TransferSingle) event.

Requirements:

- `to` cannot be the zero address.
- If the caller is not `from`, it must have been approved to spend ``from``'s tokens via [`ERC1155Upgradeable.setApprovalForAll`](#ERC1155Upgradeable-setApprovalForAll-address-bool-).
- `from` must have a balance of tokens of type `id` of at least `value` amount.
- If `to` refers to a smart contract, it must implement [`IERC1155Receiver.onERC1155Received`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155Received) and return the
acceptance magic value.

### `safeBatchTransferFrom(address from, address to, uint256[] ids, uint256[] values, bytes data)`
*public*

[Batched](ROOT:erc1155.mdx#batch-operations) version of [`ERC1155Upgradeable.safeTransferFrom`](#ERC1155Upgradeable-safeTransferFrom-address-address-uint256-uint256-bytes-).

<Callout type='warn'>
This function can potentially allow a reentrancy attack when transferring tokens
</Callout>
to an untrusted contract, when invoking [`IERC1155Receiver.onERC1155BatchReceived`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155BatchReceived) on the receiver.
Ensure to follow the checks-effects-interactions pattern and consider employing
reentrancy guards when interacting with untrusted contracts.

Emits either a [`TransferSingle`](https://docs.openzeppelin.com/contracts/api/utils#TransferSingle) or a [`TransferBatch`](https://docs.openzeppelin.com/contracts/api/utils#TransferBatch) event, depending on the length of the array arguments.

- `ids` and `values` must have the same length.
- If `to` refers to a smart contract, it must implement [`IERC1155Receiver.onERC1155BatchReceived`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155BatchReceived) and return the
acceptance magic value.

<a id="ERC1155Upgradeable-_update-address-address-uint256---uint256---"></a>

### `_update(address from, address to, uint256[] ids, uint256[] values)`
*internal*

Transfers a `value` amount of tokens of type `id` from `from` to `to`. Will mint (or burn) if `from`
(or `to`) is the zero address.

Emits a [`TransferSingle`](https://docs.openzeppelin.com/contracts/api/utils#TransferSingle) event if the arrays contain one element, and [`TransferBatch`](https://docs.openzeppelin.com/contracts/api/utils#TransferBatch) otherwise.

Requirements:

- If `to` refers to a smart contract, it must implement either [`IERC1155Receiver.onERC1155Received`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155Received)
  or [`IERC1155Receiver.onERC1155BatchReceived`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155BatchReceived) and return the acceptance magic value.
- `ids` and `values` must have the same length.

<Callout>
The ERC-1155 acceptance check is not performed in this function. See [`ERC1155Upgradeable._updateWithAcceptanceCheck`](#ERC1155Upgradeable-_updateWithAcceptanceCheck-address-address-uint256---uint256---bytes-) instead.
</Callout>

<a id="ERC1155Upgradeable-_updateWithAcceptanceCheck-address-address-uint256---uint256---bytes-"></a>

### `_updateWithAcceptanceCheck(address from, address to, uint256[] ids, uint256[] values, bytes data)`
*internal*

Version of [`ERC1155Upgradeable._update`](#ERC1155Upgradeable-_update-address-address-uint256---uint256---) that performs the token acceptance check by calling
[`IERC1155Receiver.onERC1155Received`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155Received) or [`IERC1155Receiver.onERC1155BatchReceived`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155Receiver-onERC1155BatchReceived) on the receiver address if it
contains code (eg. is a smart contract at the moment of execution).

<Callout type='warn'>
Overriding this function is discouraged because it poses a reentrancy risk from the receiver. So any
</Callout>
update to the contract state after this function would break the check-effect-interaction pattern. Consider
overriding [`ERC1155Upgradeable._update`](#ERC1155Upgradeable-_update-address-address-uint256---uint256---) instead.

<a id="ERC1155Upgradeable-_setURI-string-"></a>

### `_setURI(string newuri)`
*internal*

Sets a new URI for all token types, by relying on the token type ID
substitution mechanism
[defined in the ERC](https://eips.ethereum.org/EIPS/eip-1155#metadata).

By this mechanism, any occurrence of the `\id\` substring in either the
URI or any of the values in the JSON file at said URI will be replaced by
clients with the token type ID.

For example, the `https://token-cdn-domain/\id\.json` URI would be
interpreted by clients as
`https://token-cdn-domain/000000000000000000000000000000000000000000000000000000000004cce0.json`
for token type ID 0x4cce0.

See [`ERC1155Upgradeable.uri`](#ERC1155Upgradeable-uri-uint256-).

Because these URIs cannot be meaningfully represented by the [`URI`](https://docs.openzeppelin.com/contracts/api/token/ERC6909#URI) event,
this function emits no events.

### `uri(uint256 tokenId) ‚Üí string`
*public*

See [`IERC1155MetadataURI.uri`](https://docs.openzeppelin.com/contracts/api/token/ERC1155#IERC1155MetadataURI-uri).

This implementation returns the concatenation of the `_baseURI`
and the token-specific uri if the latter is set

This enables the following behaviors:

- if `_tokenURIs[tokenId]` is set, then the result is the concatenation
  of `_baseURI` and `_tokenURIs[tokenId]` (keep in mind that `_baseURI`
  is empty per default);

- if `_tokenURIs[tokenId]` is NOT set then we fallback to `super.uri()`
  which in most cases will contain `ERC1155._uri`;

- if `_tokenURIs[tokenId]` is NOT set, and if the parents do not have a
  uri value set, then the result is empty.
