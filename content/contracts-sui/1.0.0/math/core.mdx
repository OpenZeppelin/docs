---
title: openzeppelin_math
---

The `openzeppelin_math` package provides reusable math primitives with explicit rounding and overflow-aware behavior.

## Public Modules

- `rounding`: `RoundingMode` enum with `down`, `up`, `nearest`.
- `decimal_scaling`: decimal conversion helpers between `u64` and `u256`.
- `u8`, `u16`, `u32`, `u64`, `u128`, `u256`: width-specific arithmetic modules with the same API family.
- `u512`: wide-integer helper used for safe 512-bit intermediates in `u256` operations.

## Common API Surface

For each width module (`u8` ... `u256`), the package exposes:

- `average(a, b, rounding_mode)`
- `checked_shl(value, shift)` and `checked_shr(value, shift)`
- `mul_div(a, b, denominator, rounding_mode)`
- `mul_shr(a, b, shift, rounding_mode)`
- `clz(value)`, `msb(value)`
- `log2(value, rounding_mode)`, `log10(value, rounding_mode)`, `log256(value, rounding_mode)`
- `sqrt(value, rounding_mode)`
- `inv_mod(value, modulus)` and `mul_mod(a, b, modulus)`

Most overflow-sensitive operations return `Option<T>`, so you can enforce explicit handling.

## Examples

```move
module my_sui_app::math_examples;

use openzeppelin_math::rounding;
use openzeppelin_math::u256;
use std::option;

public fun compute_vesting_release(total: u256, elapsed: u256, duration: u256): u256 {
    let prorated = u256::mul_div(total, elapsed, duration, rounding::down());
    option::destroy_some(prorated)
}
```

```move
module my_sui_app::decimal_examples;

use openzeppelin_math::decimal_scaling;

/// Convert from 18 decimals to 9 decimals (truncating fractional residue).
public fun normalize_amount(raw_amount: u256): u64 {
    decimal_scaling::safe_downcast_balance(raw_amount, 18, 9)
}
```

## Behavior and Safety Notes

- `safe_downcast_balance` truncates when scaling down; it does not round.
- `safe_downcast_balance` aborts when result does not fit in `u64`.
- `mul_div` aborts if denominator is zero.
- `mul_mod` aborts if modulus is zero.
- `inv_mod` returns `none` when the inverse does not exist.

## `u512` in Practice

You typically do not need to use `u512` directly. It is used internally to preserve correctness for wide intermediates (for example in `u256` multiplication/division paths), but it remains public for advanced integrations.
