This page covers production-oriented integration patterns for the Sui packages in `v1.0.0`.

## `openzeppelin_access` Integration

### Two-Step Ownership Transfers

Use `openzeppelin_access::two_step_transfer` when the caller initiating the transfer is the same principal that should retain cancel authority.

Core flow:

1. Wrap a capability/object with `wrap<T>`.
2. Current owner calls `initiate_transfer(wrapper, new_owner, ctx)`.
3. New owner calls `accept_transfer(request, wrapper_ticket, ctx)`.
4. Current owner may recover control with `cancel_transfer(...)`.

```move
module my_sui_app::two_step_integration;

use openzeppelin_access::two_step_transfer;
use sui::object;
use sui::tx_context::TxContext;

public struct TreasuryCap has key, store {
    id: object::UID,
}

public fun start_transfer(
    wrapper: two_step_transfer::TwoStepTransferWrapper<TreasuryCap>,
    new_owner: address,
    ctx: &mut TxContext,
) {
    two_step_transfer::initiate_transfer(wrapper, new_owner, ctx);
}
```

<Callout type="warn">
`two_step_transfer` records `ctx.sender()` as `from` when transfer is initiated. Do not use it in shared-object custody flows where arbitrary executors can initiate transfers.
</Callout>

### Delayed Ownership Transfers

Use `openzeppelin_access::delayed_transfer` to enforce a minimum delay before transfer/unwrap execution.

Core flow:

1. Wrap with `wrap<T>(obj, min_delay_ms, ctx)`.
2. Schedule action with `schedule_transfer(...)` or `schedule_unwrap(...)`.
3. After `clock.timestamp_ms() >= execute_after_ms`, execute with `execute_transfer(...)` or `unwrap(...)`.
4. Cancel pending action with `cancel_schedule(...)`.

```move
module my_sui_app::delayed_integration;

use openzeppelin_access::delayed_transfer;
use sui::clock::Clock;
use sui::object;
use sui::tx_context::TxContext;

public struct AdminCap has key, store {
    id: object::UID,
}

public entry fun queue_transfer(
    wrapper: &mut delayed_transfer::DelayedTransferWrapper<AdminCap>,
    new_owner: address,
    clock: &Clock,
    ctx: &mut TxContext,
) {
    delayed_transfer::schedule_transfer(wrapper, new_owner, clock, ctx);
}
```

## `openzeppelin_math` Integration

### Deterministic Rounding and Overflow-Aware Arithmetic

`openzeppelin_math` exposes the same API family across `u8`, `u16`, `u32`, `u64`, `u128`, and `u256`:

- `average`, `mul_div`, `mul_shr`
- `checked_shl`, `checked_shr`
- `log2`, `log10`, `log256`, `sqrt`
- `inv_mod`, `mul_mod`

Use `rounding::down()`, `rounding::up()`, or `rounding::nearest()` to make rounding policy explicit.

### Decimal Alignment for Cross-Token/Cross-Chain Flows

`decimal_scaling` is designed for decimal conversion boundaries:

- `safe_downcast_balance(raw_amount: u256, source_decimals, target_decimals): u64`
- `safe_upcast_balance(amount: u64, source_decimals, target_decimals): u256`

Important behavior:

- Scaling down truncates fractional parts (does not round).
- Decimals must be `<= 24`.

```move
module my_sui_app::math_integration;

use openzeppelin_math::decimal_scaling;

public fun to_sui_precision(eth_amount_18_decimals: u256): u64 {
    decimal_scaling::safe_downcast_balance(eth_amount_18_decimals, 18, 9)
}
```
